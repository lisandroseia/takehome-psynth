# Document Review Queue

A full-stack internal tool for managing documents submitted for review. Built as a take-home assessment for Psynth.

---

## Project Overview

Users can view, filter, search, create, update, and delete documents in a review queue. The backend exposes a REST API with full filtering, sorting, and pagination support. The frontend provides a clean interface with real-time feedback on every action.

---

## API Endpoint Summary

| Method | Path | Description |
|---|---|---|
| `GET` | `/documents` | List documents with optional filters, search, sort, and pagination |
| `GET` | `/documents/{id}` | Get a single document by ID |
| `POST` | `/documents` | Create a new document |
| `PATCH` | `/documents/{id}` | Update document status |
| `DELETE` | `/documents/{id}` | Delete a document |

**Query parameters for `GET /documents`:**

| Param | Type | Description |
|---|---|---|
| `status` | `pending \| in_review \| approved \| rejected` | Filter by status |
| `priority` | `low \| medium \| high` | Filter by priority |
| `category` | `Legal \| Marketing \| ...` | Filter by category |
| `q` | `string` | Full-text search on title and submitter name |
| `sort_by` | `created_at \| title \| priority \| status \| submitter_name` | Sort field |
| `sort_order` | `asc \| desc` | Sort direction (default: `desc`) |
| `page` | `integer ≥ 1` | Page number (default: `1`) |
| `page_size` | `1–100` | Results per page (default: `10`) |

API documentation is auto-generated by FastAPI and available at `/docs`.

## Known Limitations

- **No persistence** — data is stored in memory and resets on every restart (intentional per spec).
- **Single worker only** — the in-memory store is not thread-safe across multiple Uvicorn workers.
- **O(n) queries** — filtering and search scan all documents on every request; no secondary indexes exist. Acceptable for the current dataset size, but would not scale to large volumes without a real database.
- **Text search is substring-based** — no stemming, ranking, or relevance scoring.
- **No authentication or authorization** — all endpoints are public.
- **No automated tests** — explicitly out of scope per the assessment brief.
- **UI design is intentionally simple** — visual polish was deprioritized in favor of architecture, correctness, and functionality.

---

## Tradeoffs Made

**Layered backend architecture (router → service → repository)**
A flat single-file approach would have been faster to write but harder to extend. Separating the three concerns means the storage layer can be swapped for a real database by only touching `repository.py`, with no changes required in the service or routing logic.

**TanStack Query instead of `useState` + `useEffect`**
Manual data fetching with `useEffect` leads to inconsistent loading/error states, no deduplication, and difficult cache invalidation. TanStack Query handles all of that declaratively and provides optimistic cache updates (e.g., invalidating the list after a create or delete) for free.

**Structured query key factory**
`documentKeys` in `api/hooks/documents.ts` centralizes all cache key definitions. This makes targeted invalidation (e.g., only invalidating lists, not detail views) explicit and maintainable.

**Generator chain for filtering**
Instead of four sequential list comprehensions (each allocating a new list), the service builds a lazy generator chain and materializes it once at the end. All active filters run in a single pass over the data, reducing intermediate memory allocations.

**Client-side validation mirrors server-side rules**
Form validation in `CreateDocumentForm.tsx` enforces the same field constraints as the Pydantic model (`title` min 3 / max 200 chars, etc.). This provides immediate feedback without an extra round-trip, while the server remains the authoritative validator.

**Debounced search**
Search input is debounced at 300ms via `useDebounce` before triggering an API call. Without this, every keystroke would fire a request.

**`page_size` capped at 100**
The API enforces a maximum page size to prevent a single request from returning the entire dataset as an unbounded payload.

**Pydantic enums for all categorical fields**
Status, priority, and category are all defined as `str` enums in Pydantic. This keeps the OpenAPI schema clean and self-documenting, and ensures invalid values are rejected at the boundary with a clear 422 error.

## AI Tool Usage

### Tools Used
- **Cursor** — initial project scaffolding
- **Claude (claude.ai)** — architectural planning and step-by-step implementation strategy
- **Claude Code** — feature implementation with frontend/backend integration

### How They Were Used

**Cursor** was used to bootstrap the project structure: generating the initial folder layout, `Dockerfile`s, base configuration files, and a first pass at several React components.

**Claude** was used as an architectural thinking partner before writing code. It helped plan the layered backend structure, identify the right tools for the frontend (TanStack Query over manual fetch hooks), and sequence the implementation to keep frontend and backend aligned throughout.

**Claude Code** was used to implement specific connected features — filters, pagination, sorting, the API client, and React Query hooks — where changes needed to be consistent across multiple files simultaneously.

### Accepted Suggestions
- TypeScript types in `types/document.ts` aligned 1:1 with the Pydantic models
- Generator chain for single-pass filtering in the service layer
- Docker and Docker Compose configuration

### Modified Suggestions
- **Layered backend architecture** — Claude's initial proposal was a basic flat structure. After the developer pushed for a more production-realistic approach, the layered router / service / repository separation was designed collaboratively and implemented with AI assistance.
- **TanStack Query** — the developer identified TanStack Query as the right tool; Cursor had generated a basic `useEffect`-based implementation which was then rewritten to use React Query correctly, including the query key factory and cache invalidation strategy.
- Several React components were restructured after generation — AI-generated components sometimes over-abstracted or missed prop types; these were rewritten to match the actual data flow.
- The initial API client lacked structured error typing; `ApiError` with a typed `body` field was added manually.

### Rejected Suggestions
- An early suggestion to use `useReducer` for filter state in `App.tsx` was rejected as unnecessary complexity — individual `useState` calls are clearer and sufficient for this scope.
- A suggestion to add optimistic updates for delete was rejected to keep the implementation honest; a refetch after mutation is simpler and less error-prone without tests to back it.
- A barrel `api/index.ts` that re-exported everything from all API modules was removed. Each component imports directly from its specific module (e.g. `api/hooks/documents`). Barrel exports hide where things actually live and create implicit coupling between features — removing them keeps imports explicit and the module boundary clear as the project grows.

### Validation Process
All AI-generated code was reviewed before acceptance: reading diffs carefully, checking type correctness in the IDE, and manually testing the affected flows in the browser. Backend changes were also tested against the FastAPI `/docs` interface to verify request/response shapes.

## Future Improvements

- Replace the in-memory store with PostgreSQL and SQLAlchemy for real persistence
- Add secondary indexes (or use DB indexes) to make filter queries O(1) instead of O(n)
- Implement full-text search using PostgreSQL `tsvector` or a dedicated search service
- Add authentication and role-based access control
- Write unit and integration tests for both the service layer and React components
- Add optimistic updates on the frontend for snappier UX on mutations
- Support multi-field filtering via URL query params so filtered views are shareable
- CI/CD pipeline for automated builds and deployments
